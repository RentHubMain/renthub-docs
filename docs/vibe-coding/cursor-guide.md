# Cursor 使用指南

[← 返回 Vibe Coding 板块](./README.md) · [← 返回文档首页](../../README.md)

## 1. 账号获取

Cursor Pro 账号直接在 **淘宝**购买，备注需要发票，店家通常可提供电子发票（个人/企业均可，建议购买前确认）。

购买后登录 [cursor.com](https://www.cursor.com) 激活，确认 Pro 状态生效后再开始使用。

---

## 2. 使用限额配置

登录 Cursor 账号后，进入 **Settings > Billing** 或 **Account**，将 Fast Request 上限调整为 **Unlimited**，避免在高频使用期间因默认配额触顶而被降速（Slow Queue）。

> Unlimited 模式下仍受套餐总额度约束，并非无限制消耗，需配合下文模型策略合理使用。

---

## 3. 模型使用原则

**本团队只使用以下两类模型，不使用其他任何模型（包括 GPT 系列、Gemini、本地模型等）：**

| 类型 | 来源 | 说明 |
|------|------|------|
| Composer 模型 | Cursor 套餐内置 | 默认首选，适合日常绝大多数任务 |
| Anthropic 模型 | Claude Sonnet / Opus | 按需切换，仅用于复杂任务 |

统一使用 Anthropic 体系的好处：模型行为一致、调试经验可复用、避免因模型特性差异导致的不可预期输出。

---

## 4. Thinking 与非 Thinking 模式的区别

部分 Anthropic 模型支持开启 **Extended Thinking（深度思考）** 模式，在输出答案前进行内部推理链。

### 4.1 非 Thinking 模式（默认）

模型直接输出结果，不展示推理过程。

- 响应速度快
- token 消耗低
- 适合：代码补全、格式修改、简单问答、文档整理等

### 4.2 Thinking 模式

模型在回答前先进行一段"内部推理"（thinking token），再给出最终答案。

- 推理更严谨，能处理多步骤、有歧义或需要权衡的问题
- token 消耗显著增加（thinking 部分本身也消耗额度）
- 响应时间更长
- 适合：复杂 bug 分析、架构决策、多条件权衡、理解复杂业务逻辑

### 4.3 选用建议

| 场景 | 推荐模式 |
|------|---------|
| 日常编码、文档编写 | Composer 模型（非 thinking） |
| 复杂逻辑、重构、跨模块分析 | Claude Sonnet（非 thinking） |
| 架构设计、高难度 bug、多方案决策 | Claude Sonnet 或 Opus（thinking） |
| 任务不确定时 | 先用非 thinking，不满意再升级 |

> Thinking 模式不是"更好的模型"，而是同一模型在耗费更多 token 换取更深入推理。不要无差别开启。

---

## 5. Composer 模型（默认首选）

Cursor Pro 套餐内含的 Composer 模型适合处理绝大多数日常任务，**应作为第一选择**，不满足需求时再按第 6 节升级。

适用场景：

- 代码补全与自动生成（Tab 补全）
- 小范围修改、重命名、格式整理
- 注释生成、变量命名建议
- 解释代码片段、查找简单 bug
- 样板代码、CRUD、配置文件
- 文档撰写与内容整理（本仓库绝大多数场景）
- 常见错误修复（lint、类型报错、import 缺失等）

---

## 6. Anthropic 模型（按需切换）

仅当任务存在以下特征时，才切换到 **Claude Sonnet** 或 **Claude Opus**：

| 模型 | 切换时机 |
|------|---------|
| **Claude Sonnet** | 复杂逻辑推理、多步骤重构、需要强代码理解能力的调试、较长上下文分析 |
| **Claude Opus** | 架构级设计决策、极复杂的多文件重构、对质量要求极高且 Sonnet 多次失败的任务 |

切换判断参考：

- Composer 模型已尝试 **2 次以上**仍无法正确理解意图或输出存在明显缺陷
- 任务涉及**跨多个模块的复杂依赖关系**分析
- 需要对**业务逻辑作出架构建议**（而非只是写代码）
- 生成结果将直接进入 Code Review，对准确性要求高

不应切换的场景：

- 任务本身简单，只是因为描述不清楚导致模型误解（先优化提示词）
- 觉得 Opus 更"高级"就默认用它（浪费额度）
- 重复性、模板化的任务

## 7. 控制 Token 消耗

不合理的上下文会快速消耗 token，以下是常见控制手段：

### 7.1 精简上下文

- 只 `@` 真正相关的文件或代码块，避免把整个项目附加进去
- 使用 `@file` 精确引用，而非 `@folder`（尤其是大目录）
- 对话窗口变长后，必要时 **New Chat** 重开，避免历史记录堆积

### 7.2 编写清晰的提示词

- 描述目标而非步骤：说"把这个函数改成支持分页"而非一步步指令
- 限定范围："只修改这个文件的 X 函数，不要动其他地方"
- 明确格式要求：如"只输出修改后的代码，不需要解释"，减少无效输出

### 7.3 Agent / Composer 使用注意

- 不要开启过长的 Agent 任务链（多步 tool call）去处理简单问题
- 每次 Composer 任务完成后检查实际改动范围，防止模型"顺手"修改非目标文件
- 复杂任务拆成小步骤逐步完成，而非一次性提交大范围修改

### 7.4 及时清理

- 对话历史过长时及时归档/关闭，上下文窗口越长每轮消耗越高
- Rules 配置精简，只保留真正有效的约束，冗长的 rules 本身也消耗 token

## 8. 常见误区

| 误区 | 正确做法 |
|------|---------|
| 使用 GPT / Gemini 等非 Anthropic 模型 | 只用 Composer 模型或 Claude Sonnet / Opus |
| 遇到问题直接切 Opus | 先检查提示词是否清晰，再升级模型 |
| 无差别开启 Thinking 模式 | 仅在复杂推理场景开启，普通任务无需开启 |
| `@` 整个项目文件夹 | 只 `@` 需要的文件或片段 |
| 用一个超长对话完成所有任务 | 按任务拆分，必要时 New Chat |
| 让模型"随便重构一下" | 明确告知范围、目标与约束 |
| Unlimited = 无限制消耗 | Unlimited 指不限单次 fast request 数量，总额度仍有限 |
